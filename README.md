# Events API

## Описание
REST API для управления событиями (создание, чтение, обновление, удаление).

## Запуск проекта

1. Установите .NET SDK (версия 6.0 или выше).
2. Клонируйте репозиторий.
3. Выполните:
   ```
   dotnet build
   dotnet run
   ```
4. API будет доступен по адресу: `http://localhost:port/api/events`
5. Swagger UI: `http://localhost:port/swagger`

## Эндпоинты

| Метод | URL                | Описание                          |
|-------|--------------------|-----------------------------------|
| GET   | /api/events         | Получить список всех событий      |
| GET   | /api/events/{id}   | Получить событие по id            |
| POST  | /api/events         | Создать новое событие             |
| PUT   | /api/events/{id}   | Обновить событие                  |
| DELETE| /api/events/{id}   | Удалить событие                   |

## Параметры фильтрации для GET /api/events
Все параметры являются необязательными и могут комбинироваться.

| Параметр | Тип                | Описание                          |
|----------|--------------------|-----------------------------------|
| title    | string             | Поиск по названию (частичное совпадение, без учета регистра).|
| from     | DateTime (строка)  | События, которые начинаются не раньше указанной даты. Формат: 2026-04-22T10:00:00.|
| to       | DateTime (строка)  | События, которые заканчиваются не позже указанной даты. Формат: 2026-04-22T18:00:00.|

## Параметры пагинации для GET /api/events
| Параметр | Тип                | Значение по умолчанию| Описание                          |
|----------|--------------------|----------------------|-----------------------------------|
| page     | int                | 1                    | Номер страницы, которую необходимо вернуть |
| pageSize | int                | 10                   | Количество элементов на одной странице |

## Валидация

- Title, StartAt, EndAt, TotalSeats — обязательные поля.
- EndAt должно быть позже StartAt.
- Ответ при ошибке: В случае нарушения правил валидации API возвращает HTTP-статус 400 Bad Request с телом ответа в формате Problem Details (RFC 7807).

Пример ответа об ошибке:
{
  "type": "about:blank",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "detail": "Дата окончания (EndAt) должна быть позже даты начала (StartAt)."
}

## Сервис бронирований (Booking Service)
В дополнение к управлению событиями, API предоставляет функционал для создания и управления бронями.

1. Реализация сущности Booking
Создана модель Booking со следующими полями:

Id (Guid): Уникальный идентификатор брони.
EventId (Guid): Идентификатор события, на которое создана бронь.
Status (BookingStatus): Текущий статус брони.
CreatedAt (DateTime): Дата и время создания брони.
ProcessedAt (DateTime?): Дата и время обработки брони (заполняется фоновым сервисом).
Реализовано перечисление BookingStatus со значениями:

Pending: Бронь создана и ожидает обработки.
Confirmed: Бронь подтверждена.
Rejected: Бронь отклонена.
При создании новой брони ей автоматически присваивается уникальный Id, статус Pending и текущее время в поле CreatedAt.

2. Эндпоинты API
Создание брони
POST /api/events/{id}/book

Создает новую бронь для события с указанным id.

Успешный ответ (202 Accepted):
Тело ответа: Возвращает объект созданной брони (со статусом Pending).
Заголовок Location: Содержит ссылку на ресурс новой брони (/api/bookings/{bookingId}).
Ошибка (404 Not Found):
Возвращается, если событие с указанным id не найдено.
Получение информации о брони
GET /api/bookings/{id}
Ошибка (409 Conflict):Возвращается, если на событие нет свободных мест.
Возвращает текущее состояние брони по её идентификатору.

Успешный ответ (200 OK):
Возвращает объект брони с актуальным статусом.
Ошибка (404 Not Found):
Возвращается, если бронь с указанным id не найдена.
3. Фоновая обработка
В системе реализован фоновый сервис (BackgroundService), который автоматически обрабатывает созданные брони.

Сервис периодически проверяет хранилище на наличие броней со статусом Pending.
Для каждой найденной брони имитируется задержка (обращение к внешней системе).
После задержки статус брони автоматически меняется на Confirmed, а поле ProcessedAt заполняется текущей датой и временем.
Поведение для проверки:

Создайте бронь через POST /api/events/{eventId}/book. Статус будет Pending.
Подождите несколько секунд (больше, чем задержка в фоновом сервисе, обычно 5-7 секунд).
Запросите бронь через GET /api/bookings/{bookingId}. Вы увидите, что статус изменился на Confirmed, а поле processedAt теперь заполнено.

Примитивы синхронизации и защита от овербукинга
Для обеспечения корректной работы при одновременных запросах в сервисе используются примитивы синхронизации:

lock: Используется в методе создания брони (CreateBookingAsync). Он гарантирует, что проверка наличия свободных мест и их последующее уменьшение происходят как единая атомарная операция.
SemaphoreSlim: Используется в фоновом сервисе обработки броней. Он защищает запись в хранилище, чтобы несколько потоков не пытались одновременно обновить одну и ту же бронь или событие.
Эти механизмы предотвращают состояние гонки (race condition), когда два одновременных запроса могут "увидеть" одно и то же свободное место и обе забронировать его, что привело бы к овербукингу.

Запуск и настройка приложения
1. Требования
Для работы приложения требуется установленный PostgreSQL версии 12 или выше.

2. Настройка строки подключения к базе данных
Приложение использует строку подключения с именем DefaultConnection. Вы можете указать её одним из следующих способов:

Через переменную окружения:
Установите переменную ASPNETCORE_ENVIRONMENT=Development (для режима разработки) и задайте строку подключения.

В файле конфигурации:
Отредактируйте файл appsettings.json, заменив значение в разделе ConnectionStrings:

json
{
  "ConnectionStrings": {
    "DefaultConnection": "Host=localhost;Port=5432;Database=eventapi;Username=postgres;Password=your_password"
  },
  // ... другие настройки
}
3. Инициализация базы данных
Схема базы данных и таблицы (events, bookings) создаются автоматически при первом запуске приложения.

Приложение использует метод db.Database.EnsureCreated().
Если база данных или таблицы отсутствуют, они будут созданы на основе ваших моделей.
Примечание: Данный метод не поддерживает миграции Entity Framework Core. Если вы планируете использовать миграции, замените вызов EnsureCreated() на db.Database.Migrate().

🧪 Тестирование
Использование InMemory-провайдера
Для выполнения модульных тестов приложение использует InMemory-провайдер Entity Framework Core.

Это позволяет запускать тесты без необходимости подключения к реальной базе данных PostgreSQL.
Каждая тестовая сессия использует изолированную базу данных в оперативной памяти, что гарантирует чистоту и независимость тестов друг от друга.
